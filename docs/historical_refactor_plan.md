# Decouple 1D/1H Historical Data Refresh into a Standalone Process

## Context

`main.py` (PyQt5 dashboard) currently runs the "Update 1D Data" and "Update 1H Data"
refreshes as in-process `QThread` workers (`RefreshWorker` / `HourlyRefreshWorker` in
`src/ui/workers.py`). The user iterates on the UI code frequently and has to restart
`main.py` often — restarting kills any in-flight refresh, losing progress and forcing
a full re-run next time.

Goal: move the actual fetch work into a standalone script (`historical.py`) that runs
as an independent OS process, so closing/reopening `main.py` has zero effect on it.
`main.py` launches it, polls its status, and — per the user's explicit safety
requirement — lets them terminate it from the same button (with a confirmation
prompt), while guaranteeing already-fetched data is never lost/corrupted and reruns
only fetch what's missing.

Verified via code research (not assumed): `refresh_universe_history_to_db` (1D) and
`refresh_universe_hourly_history_to_db` (1H) in `src/utils/db_loader.py` already commit
each batch immediately (`engine.begin()` + `INSERT ... ON DUPLICATE KEY UPDATE` keyed on
`(symbol, date, interval)` / `(symbol, timestamp, source)`), and per-symbol "what's
already fetched" detection already exists. So killing the process mid-run only ever
loses the single in-flight (never-committed) batch — everything before it is durably
saved, and the next run correctly detects and skips what's done. This means the safety
requirement is satisfied by the existing DB layer; the new work is entirely about the
process boundary, status reporting, and terminate UX. `db_loader.py`/`data_loader.py`
have zero PyQt5 dependency, so `historical.py` can import them directly with no Qt cost.

**Decisions confirmed with user:**
- Remove `RefreshWorker`/`HourlyRefreshWorker` entirely (no legacy fallback).
- 1D and 1H become fully independent, simultaneously-runnable processes (drop today's
  partial serialization).

## Architecture

### 1. New file: `historical.py` (repo root)

Standalone CLI, no Qt import:

```
historical.py --mode {1d,1h} [--backfill] [--universe-limit N] --run-id UUID
```

- `--mode 1d`: `refresh_universe_history_to_db(tickers, engine, period="1y", interval="1d", ...)`
  → `refresh_chart_indicators_to_db(...)` → `refresh_scanner_metrics_to_db(...)` — same
  sequence as today's `RefreshWorker` daily branch (no hourly leg; the button never
  triggers that today either).
- `--mode 1h`: `refresh_universe_hourly_history_to_db(tickers, engine, full_period="730d", backfill=<flag>, ...)`.
- `tickers` built via `get_default_universe(max_symbols=universe_limit, refresh=True)`,
  SPY-first-deduped, same as current handlers.
- DB engine via `init_mysql_engine()` — picks up MySQL config from `.env`/`os.environ`
  automatically (`src/utils/config.py`), no special wiring needed even with a different
  CWD.
- Acquires a per-mode exclusive lock via `msvcrt.locking()` on `data/refresh_lock_1d.lock`
  / `data/refresh_lock_1h.lock` (stdlib, no new dependency; OS releases it automatically
  even on a hard kill — this is the authoritative crash-safe "am I already running"
  check). If already locked, print + exit(2) without touching the status file.
- Wires `progress_callback`/`log_callback` (existing signatures in `db_loader.py`, e.g.
  `progress_callback(symbol, processed, total, percent, eta_text)`) to a small
  `RunState` object that throttles and writes status to JSON via the existing
  `save_json()`/`load_json()` atomic-write helpers already in `src/utils/storage.py`
  (temp-file-then-replace pattern — reuse, don't reinvent).
- Writes `status="running"` immediately on start (with `pid`, `run_id`), updates
  `phase`/`progress`/`recent_log` as it goes, and `status="completed"` or `status="error"`
  (with `result.error_message`) in a `try/except/finally` on exit. Releases the lock in
  `finally`.
- stdout/stderr redirected by the launcher to `data/logs/historical_1d.log` /
  `historical_1h.log` (overwritten each run) for postmortem debugging, since the process
  is fully detached (no console).

### 2. Status file schema — `data/refresh_status_1d.json` / `_1h.json`

One independent file per mode:

```jsonc
{
  "schema_version": 1,
  "mode": "1d",
  "status": "running",           // idle (file absent) | running | completed | error | terminated
  "run_id": "3f7a1e2c...",       // uuid4 hex, generated by main.py, passed via --run-id
  "pid": 18420,
  "started_at": "2026-07-04T13:05:11+00:00",
  "updated_at": "2026-07-04T13:07:42+00:00",
  "finished_at": null,
  "backfill": false,
  "phase": "daily_history",      // 1d: starting -> daily_history -> chart_indicators -> scanner_metrics -> completed
                                  // 1h: starting -> hourly_history -> completed
  "progress": {"processed": 340, "total": 512, "percent": 66, "eta_text": "2m 15s", "current_symbol": "NVDA"},
  "recent_log": ["[13:07:42] Batch 5/6 1d: period=1mo, symbols=NVDA, TSLA, ..."],
  "result": {"updated_count": 480, "error_message": null}
}
```

### 3. New module: `src/services/historical_refresh_control.py`

Centralizes subprocess/status logic so UI handlers stay thin:

- `read_status(mode)` — tolerant `load_json` wrapper, defaults to `status="idle"`.
- `is_process_alive(pid)` — Windows liveness check via `tasklist /FI "PID eq {pid}" /FO CSV /NH`
  (no `psutil` dependency in this repo).
- `is_refresh_running(mode) -> (bool, status_dict)` — `status=="running"` AND PID alive.
- `reconcile_stale_status(mode)` — if status says running but PID is dead, atomically
  rewrite to `status="error"` (can't distinguish a crash from a manual Task Manager
  kill, so both self-heal to `"error"`, not `"terminated"`).
- `launch_refresh(mode, backfill=False, universe_limit=None)` — generates `run_id`,
  `subprocess.Popen([sys.executable, "historical.py", "--mode", mode, ...], cwd=REPO_ROOT,
  creationflags=CREATE_NEW_PROCESS_GROUP | DETACHED_PROCESS | CREATE_NO_WINDOW,
  stdin=DEVNULL, stdout=log_fh, stderr=STDOUT)`. Does **not** pre-write the status file
  (avoids a race where main.py reads a "running" record before the child has actually
  started and reconciles it away) — UI shows an optimistic "starting..." state until the
  first real poll tick sees `status="running"`.
- `terminate_refresh(mode) -> bool` — `taskkill /PID <pid> /T /F`, poll `is_process_alive`
  for ~3s to confirm death, then atomically write `status="terminated"`. Force-kill is
  safe here specifically because durability comes from the DB upsert layer, not from a
  graceful in-process handler — no need for a softer signal-based shutdown first.

### 4. `main_window.py` changes

- Remove `self.refresh_worker`/`self.hourly_refresh_worker` (and their entries in
  `closeEvent`'s running-workers-block-close list — the whole point of this feature is
  that closing main.py must not affect the refresh, so it must not be blocked by it
  either).
- Add a `QTimer` (`_refresh_poll_timer`, ~2s interval) that always runs, calling
  `_poll_refresh_status()`.
- At startup: call `reconcile_stale_status()` for both modes, then one synchronous
  `_poll_refresh_status()` before `show()`, so reopening main.py mid-refresh immediately
  reflects the correct running/terminate-mode state (satisfies the "detect an
  already-running historical.py after main.py restart" requirement).

### 5. `scanner_mixin.py` — button handlers become start/terminate toggles

Replace `refresh_data_to_db` / `refresh_hourly_data_to_db` bodies:

```python
def refresh_data_to_db(self) -> None:
    if not self.db_enabled:
        QMessageBox.warning(...); return
    running, _ = is_refresh_running(MODE_1D)
    if running:
        self._confirm_and_terminate(MODE_1D, "1D Data")
        return
    launch_refresh(MODE_1D, universe_limit=self.universe_limit)
    self._poll_refresh_status()
```

(1H handler is structurally identical, no cross-mode check per the concurrency
decision above.)

`_confirm_and_terminate(mode, label)` shows `QMessageBox.question(..., default=No)`:
*"A {label} refresh is currently running. Are you sure you want to terminate it? Data
already saved to MySQL for symbols processed so far will NOT be lost. The next refresh
will resume from where this one left off."* On Yes → `terminate_refresh(mode)`.

`_poll_refresh_status()` updates button text (`"Update 1D Data"` ↔
`"Terminate 1D Update (66%, ETA 2m 15s)"`), progress bar/label, appends only *new*
`recent_log` lines to the log panel, and on a terminal status transition (de-duped by
`finished_at`) calls a generic `_handle_refresh_terminal_status()` that reuses the
existing `show_refresh_complete(count)` / `show_refresh_error(message)` /
`update_dashboard_summary(force=True)` — none of which need signature changes, since
they already only take a count/message and re-derive UI state from the DB rather than
from worker output.

Remove now-obsolete: `_is_refresh_worker_running`, `_is_hourly_refresh_worker_running`,
`_is_daily_update_complete`, `_sync_refresh_button_states`, `_on_refresh_worker_finished`,
`_on_hourly_refresh_worker_finished`, `_on_daily_data_update_finished`,
`_on_refresh_finished`, `_on_refresh_error`, `_on_hourly_refresh_finished`,
`_on_hourly_refresh_error`.

### 6. `src/ui/workers.py`

Delete `RefreshWorker` and `HourlyRefreshWorker` classes entirely.

### 7. `tests/test_refactor_boundaries.py`

Drop `RefreshWorker` from the import and the `assert RefreshWorker is not None` line
(keep `ScannerWorker`, `WatchlistAiWorker`).

### 8. Misc safety items to check while implementing (not design changes)

- Add `data/refresh_status_*.json` and `data/refresh_lock_*.lock` to `.gitignore`

## Safety hardening (follow-up review)

A follow-up review of the initial implementation tightened several correctness
edges before treating this as production-safe. All of the following live in
`src/services/historical_refresh_control.py` unless noted:

- **`terminate_refresh()` no longer assumes success.** It calls `taskkill`,
  waits up to `wait_seconds`, then re-checks `is_process_alive(pid)`. It only
  writes `status="terminated"` when the PID is confirmed dead *and* that death
  can be attributed to this request. If the process is still alive, the status
  is kept/restored to `"running"` with `result.error_message = "Termination
  requested but process is still running."` and the function returns `False`.
  If the child happened to finish or error on its own in the same window (a
  race between our kill and its natural completion), its authentic
  `completed`/`error` status is left untouched rather than being relabeled
  `terminated`.
- **Status is now `run_id`-aware with an explicit `starting` state.**
  `"starting"` is a *parent*-written, transient state: `launch_refresh()`
  writes it (with the real PID and a fresh `run_id`) immediately after
  `Popen()` returns, before the function returns to its caller. This means a
  stale terminal status from a previous run_id is never the freshest thing on
  disk by the time the UI's next poll runs. `historical.py` overwrites the
  same run_id's record to `"running"` shortly after.

  `Popen()` returning control to the parent is not synchronized with the
  child's own execution — a fast-starting (or fast-failing) child can write
  its own `running`/`completed`/`error`/`terminated` status *before* the
  parent gets around to writing `"starting"`. To avoid clobbering that real
  status, `launch_refresh()` re-reads the status file right before writing and
  skips the `"starting"` write entirely if a status for the same `run_id`
  already exists (see the "race hardening" note below).

  A live PID in `"starting"` is always considered active — `is_refresh_running()`
  no longer ages `"starting"` out after a fixed window, since a child that is
  alive but merely slow to flip its own status to `"running"` is not
  meaningfully different from one already reporting `"running"`.
  `reconcile_stale_status()` only converts `"starting"` (or `"running"`) to
  `"error"` when the PID is confirmed dead.
- **The UI no longer replays stale terminal events.** `scanner_mixin.py`
  tracks `_refresh_active_run_id` (the run_id it's currently watching per
  mode) and `_refresh_last_finished_at` (the last `finished_at` already
  handled per mode). `_is_new_terminal_refresh_event()` only fires the
  completion/error dialog and `update_dashboard_summary()` when the event's
  `run_id` matches the currently-tracked run and its `finished_at` hasn't been
  seen before. At startup, `main_window.py` seeds `_refresh_last_finished_at`
  from whatever's already on disk (after reconciling any stale
  running/starting record) so a completed/error/terminated status left over
  from a previous session is shown passively in the progress label but does
  not pop a dialog every time the app opens.
- **Interrupted 1D refreshes clearly flag stale derived data.** The status
  schema (bumped to `schema_version: 2`) adds `completed_phases` (a list built
  up as `daily_history` → `chart_indicators` → `scanner_metrics` each finish)
  and `derived_data_complete` (true only once all phases required for that
  mode are done — for 1H, just `hourly_history`). If a 1D refresh is
  terminated or errors after `daily_history` but before the derived phases
  finish, the UI appends a note: *"Price data was saved, but chart
  indicators/scanner metrics did not finish. Run the 1D refresh again to bring
  derived data back in sync."* `status="completed"` still only means all
  required phases finished.
- **`is_process_alive()` parses `tasklist` output properly** via `csv.reader`
  instead of substring matching, comparing the PID column as an integer. This
  fixes a false-positive class of bug where PID `123` would match within a
  PID field like `1234`, and cleanly returns `False` for the no-match
  (`"INFO: No tasks..."`) and malformed-output cases.
- **Runtime state is no longer committed.** `data/refresh_status_*.json`,
  `data/refresh_status_*.json.bak`, `data/refresh_lock_*.lock`, and
  `data/logs/` are gitignored. The pre-existing `data/*.json.bak` backup files
  (written automatically by `src/utils/storage.py:save_json`) are untracked
  from git entirely, since they are runtime-generated, not source data.
- **Tests**: `tests/test_historical_refresh_control.py` covers
  `read_status`/`reconcile_stale_status`/`terminate_refresh`/`launch_refresh`
  behavior (with `subprocess.run`/`Popen`/`is_process_alive` monkeypatched —
  no real process spawned, no MySQL/yfinance touched) and the UI's terminal-
  event de-duplication logic in isolation.

### Follow-up: parent/child status-write race

`launch_refresh()` writing `"starting"` after `Popen()` returns is not
synchronized with the child actually running — a fast-starting or
fast-failing `historical.py` could write its own real status for the same
`run_id` *before* the parent got to write `"starting"`, and the parent would
then clobber that real status (potentially a terminal `completed`/`error`)
with a stale `"starting"`. Fixed by re-reading the status right before that
write and skipping it entirely if a same-`run_id` status already exists in
`{"running", "completed", "error", "terminated"}`. Combined with dropping the
`STARTING_STATUS_MAX_AGE_SECONDS` liveness gate (a live PID in `"starting"` is
now always active, never aged out), `"starting"` is purely a best-effort
optimistic UI hint — the source of truth is always whatever the child last
wrote, and PID liveness. See `tests/test_historical_refresh_control.py` for
the regression tests (`test_launch_refresh_does_not_overwrite_child_*`,
`test_starting_with_live_pid_is_running_regardless_of_age`).
